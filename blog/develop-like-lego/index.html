พัฒนาระบบโดยคิดถึง Lego

28 สิงหาคม 2019

ถ้าคุณทำงานมาหลาย Project คุณคงต้องเคยทำงานแล้วพบว่า ทำไมมีงานบางงานที่ทำซ้ำเหมือนเดิม ถึงต้องเขียน Code ซ้ำไปซ้ำมาเหมือนเดิมด้วยละ ทำไม Code แบบนี้ถึงต้องถูกเขียนยาวๆแบบนี้ทุกไฟล์ เวลาจะทำอะไรแบบนี้ด้วย

สิ่งเหล่านี้จะเกิดขึ้นมา เวลาที่คุณพบว่า การออกแบบ Code เป็นการทำงานแบบที่ไม่มีการแคร์สิ่งที่เรียกว่า “Don’t repeat yourself” หรือเจ้า DRY หรือแปลแบบตรงๆอย่างคนอ่านภาษาอังกฤษไม่ออกอย่างผมก็ อย่าทำตัวเองซ้ำ นั้นแหละ 555

มันมักจะเกิดจากความคิดแค่ว่า อ่อ...การทำสิ่งนี้ คือต้องเขียนด้วย Code แบบนี้ Copy จากไฟล์นั้นมาวาง เปลี่ยนตัวแปรซะ ปรับข้อมูลให้ตรงกับตรงนี้ แบบนี้ก็จบแล้ว เราเลยไม่ได้คิดไปเลยกว่านั้น

พอเราย้อนกลับไปดู เราจึงพบว่า สิ่งที่เราเคยทำลงไป มันเป็นการทำงานซ้ำอยู่กับที่ ทำงานเดิมๆ เขียน Code เดิมๆที่ถ้าคุณทำงานมานานจะโคตรเบื่อเลย เพราะคุณจะเห็น Code ที่ยาวๆแบบนั้นเต็มไปหมด

ทีนี้วันนี้ผมมีวิธีคิดมานำเสนอครับ งั้นเรามาลองคิดใหม่ ที่ระบบของเราแก้ไขยาก ที่ระบบเราพัฒนาล่าช้า ที่ระบบเราไม่ยืดหยุ่น อันที่จริงมันกก็มาจากสิ่งเหล่านี้นั้นแหละครับ

เพราะเราไปมัวทำซ้ำในสิ่งที่เราไม่จำเป็นอยู่บ่อยๆ ลองคิดว่าระบบคุณก็คือ บ้านหนึ่งหลัง ที่คุณทำเป็น Model จำลองขึ้นมาจากฟองน้ำ จากไม้ จากกระดาษแข็ง ตัดเป็นแผ่นๆใหญ่มาแปะ มาต่อกัน วันหนึ่งคุณบอกว่า อยากเจาะผนังเพิ่มหน้าบ้าน อ้าว...ก็เราใช้แผ่น 1 แผ่นเป็นผนัง แล้วเจาะแล้วมันจะง่ายหรอ จะปลอดภัยหรอ ไม่กระทบส่วนอื่นๆหรอ

กับลองคิดว่า บ้านของคุณเป็น Lego คุณต้องต่อแต่ละอย่างจากชิ้นเล็กๆทีละชิ้นจนได้ผนังบ้าน แล้วก็ได้หลังคา ได้เสา ได้ประตู วันหนึ่งคุณอย่างเจาะผนัง คุณก็แค่ยกส่วนที่จะเอาออกมาแกะ แล้วคุณก็จะได้หน้าต่างทันที ได้ไม่ส่งผลกับคนอื่น

แต่ลองมาปรับแนวคิดครับ เปลี่ยนเป็นมอง Code ให้เป็นแค่เพียง function ทำงานเล็กๆ เป็นอิสระต่อกัน ใครจะเอาไปใช้เมื่อไหร่ก็ได้ เช่นถ้าคุณต้องการข้อมูลจากฐานข้อมูลซักตารางแบบ all เลย เช่น ข้อมูลลูกค้าดึงทุกอย่างมาหมดเลย

ปรกติคุณอาจจะทำ getAll() ใน customerModel เพื่อ select * from customer ทีนี้พอบอกว่า อ่อ...อยากได้แบบนี้กับ vendor ด้วย คุณจะทำไง คุณก็อาจจะไปสร้าง getAll() ใน vendorModel แล้วเขียนเหมือนกัน เห็นไหม ได้เหมือนกันเลย ได้ซ้ำเหมือนกันเดะเลย 555

แต่พอเรามอง Code ของเราเป็น Lego เป็นชิ้นเล็ก ชิ้นน้อย เป็นอิสระจากกัน แทนที่ Code เหล่านี้คุณก็สร้างเป็น getModel ขึ้นมาแทน ส่วนในนั้นคุณก็ all(ชื่อตาราง) ลงไปแบบนี้แทน

select * from $table

แบบนี้คุณเขียนครั้งเดียวตลอดชีวิต

การมอง Code เป็นชิ้นเล็กชิ้นน้อย มันจะทำให้คุณทำงานได้เร็วขึ้น แก้ไขงานง่ายขึ้น ทำงานร่วมกันเป็นทีมดีขึ้น พัฒนางานใหม่ในระบบเดิมง่ายขึ้น เข้าใจ code ง่ายขึ้น clean code มากขึ้น ลดความจำเจในชีวิตอีกด้วย 555

อันที่จริงแล้ว แนวคิดในการเขียน code มีหลากหลายเต็มไปหมด แต่ละอย่างก็ดีหมดนั้นแหละ แต่ผมชอบการที่เราทำงานเล็กๆ และไม่ทำงานซ้ำ เมื่อไหร่ที่รู้ว่าจะต้องเขียนแบบนี้ มากกว่า 1 ครั้ง ผมจะไปเขียนเป็น function จะเรียกว่าเป็น library หรือ helper หรืออะไรก็สุดแล้วแต่ แต่มันจะเอามาทำงานได้สะดวกมาก

การเขียน code เล็กๆแบบนี้ ไม่ได้บอกว่าประสิทธิภาพของระบบจะเร็ว แต่อย่างใดนะครับ มันเกี่ยวข้องกับเรื่องของการพัฒนาเท่านั้นเอง

ถ้าเห็นว่าเรื่องที่คุณเสียเวลาอ่านมาถึงบรรทัดนี้พอเป็นประโยชน์บ้าง ลองกลับไปแล้วเริ่มต้น code ของคุณใหม่ให้เล็กเป็นชิ้นๆ แล้วสามารถหยิบไปต่อกับอะไรก็เป็นรูปเป็นร่างได้เสมอดูครับ
